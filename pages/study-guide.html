🗄️ SQL Syntax Cheat Sheet

Transaction Control

Transactions ensure ACID properties. Always use BEGIN, COMMIT, and ROLLBACK for multi-step operations.

SQL 
-- Start a transaction
BEGIN;
-- or
BEGIN TRANSACTION;
-- or
START TRANSACTION;

-- Perform operations
UPDATE Account SET balance = balance - 100 WHERE account_id = 1;
UPDATE Account SET balance = balance + 100 WHERE account_id = 2;

-- Save changes permanently
COMMIT;

-- Or undo all changes
ROLLBACK;

-- Create a savepoint (partial rollback point)
SAVEPOINT my_savepoint;

-- Rollback to savepoint
ROLLBACK TO SAVEPOINT my_savepoint;
💡 Pro Tip

Always pair related operations in a transaction. A bank transfer that debits one account MUST credit another - use transactions to ensure both succeed or both fail.

UNION / UNION ALL

Combine results from multiple SELECT statements. Columns must be compatible in number and type.

SQL 
-- UNION removes duplicates
SELECT emp_name, salary FROM Employees_US
UNION
SELECT emp_name, salary FROM Employees_UK
ORDER BY salary DESC;

-- UNION ALL keeps duplicates (faster)
SELECT product_name FROM Current_Products
UNION ALL
SELECT product_name FROM Discontinued_Products;

-- Multiple SELECTs
SELECT col1, col2 FROM TableA
UNION
SELECT col1, col2 FROM TableB
UNION ALL
SELECT col1, col2 FROM TableC;
⚠️ Important

Column names in the result come from the first SELECT. ORDER BY can only appear at the end (in PostgreSQL) and applies to the final combined result.

Common Table Expressions (CTEs)

Named temporary result sets that make complex queries more readable. Support recursive queries for hierarchical data.

Non-Recursive CTE 
-- Basic CTE syntax
WITH CTE_Name AS (
    SELECT column1, column2
    FROM SomeTable
    WHERE condition
)
SELECT * FROM CTE_Name;

-- Multiple CTEs
WITH 
    Sales AS (
        SELECT emp_id, SUM(amount) as total_sales
        FROM Orders
        GROUP BY emp_id
    ),
    TopSales AS (
        SELECT emp_id, total_sales
        FROM Sales
        WHERE total_sales > 10000
    )
SELECT e.emp_name, t.total_sales
FROM Employees e
JOIN TopSales t ON e.emp_id = t.emp_id;
Recursive CTE 
-- Recursive CTE for org chart
WITH RECURSIVE OrgHierarchy AS (
    -- Anchor member: start with top-level employees
    SELECT emp_id, emp_name, manager_id, 0 as level
    FROM Employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    -- Recursive member: find subordinates
    SELECT e.emp_id, e.emp_name, e.manager_id, oh.level + 1
    FROM Employees e
    JOIN OrgHierarchy oh ON e.manager_id = oh.emp_id
)
SELECT * FROM OrgHierarchy ORDER BY level, emp_name;

-- Recursive CTE for bill of materials
WITH RECURSIVE BOM AS (
    SELECT component_id, quantity as total_qty, 1 as level
    FROM Components WHERE product_id = ?
    
    UNION ALL
    
    SELECT c.component_id, c.quantity * b.total_qty, b.level + 1
    FROM Components c
    JOIN BOM b ON c.product_id = b.component_id
)
SELECT * FROM BOM;

Triggers

Automatically execute functions when specific database events occur.

Trigger Example 
-- Create trigger function
CREATE OR REPLACE FUNCTION log_employee_changes()
RETURNS TRIGGER AS $$
BEGIN
    IF (TG_OP = 'DELETE') THEN
        INSERT INTO AuditLog (table_name, operation, old_data, changed_at)
        VALUES ('employees', 'DELETE', row_to_json(OLD), NOW());
        RETURN OLD;
    ELSIF (TG_OP = 'UPDATE') THEN
        INSERT INTO AuditLog (table_name, operation, old_data, new_data, changed_at)
        VALUES ('employees', 'UPDATE', row_to_json(OLD), row_to_json(NEW), NOW());
        RETURN NEW;
    ELSIF (TG_OP = 'INSERT') THEN
        INSERT INTO AuditLog (table_name, operation, new_data, changed_at)
        VALUES ('employees', 'INSERT', row_to_json(NEW), NOW());
        RETURN NEW;
    END IF;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Create trigger
CREATE TRIGGER employee_audit_trigger
    AFTER INSERT OR UPDATE OR DELETE ON Employees
    FOR EACH ROW
    EXECUTE FUNCTION log_employee_changes();

-- Drop trigger
DROP TRIGGER IF EXISTS employee_audit_trigger ON Employees;
Keyword Meaning Usage
NEW New row values INSERT, UPDATE operations
OLD Old row values UPDATE, DELETE operations
TG_OP Trigger operation INSERT, UPDATE, DELETE
BEFORE Fire before change Validation, transformation
AFTER Fire after change Logging, cascading changes

Indexes

Speed up queries at the cost of slower writes and storage space.

Index Examples 
-- Basic index
CREATE INDEX idx_employee_name ON Employees(last_name);

-- Unique index
CREATE UNIQUE INDEX idx_employee_email ON Employees(email);

-- Composite index (column order matters!)
CREATE INDEX idx_employee_dept_salary ON Employees(dept_id, salary);

-- Index with INCLUDE (covering index)
CREATE INDEX idx_employee_dept ON Employees(dept_id) 
INCLUDE (emp_name, salary);

-- Analyze query performance
EXPLAIN ANALYZE 
SELECT * FROM Employees WHERE dept_id = 5;

-- Drop index
DROP INDEX IF EXISTS idx_employee_name;

-- Cluster table (physically reorder)
CLUSTER Employees USING idx_employee_dept;
📊 When to Index
Columns frequently in WHERE, JOIN, ORDER BY. High cardinality columns.
⚠️ When NOT to Index
Small tables, frequently updated columns, low cardinality columns.
💡 Covering Index
Index includes all columns needed for a query - no table lookup needed.

👥 Subtype Relationships

🎯 The "Is A" Test
If "A is a B" makes sense and every attribute of B applies to A, then A can be a subtype of B.

Example: "Student is a Person" ✓
"Car is a Person" ✗
⊗ Exclusive (Disjoint)
An instance can belong to only one subtype.

Example: An Employee is either Salaried OR Hourly, never both.
○ Overlapping
An instance can belong to multiple subtypes.

Example: A Person can be both a Student AND an Employee.

🔗 Weak Entities & Identifying Relationships

A weak entity cannot be uniquely identified by its own attributes alone. It depends on a strong entity (owner) for identification through an identifying relationship.
Characteristic Description Example
Identification Needs owner's PK + partial key ORDER_LINE needs ORDER_ID + line_number
Existence Cannot exist without owner No ORDER_LINE without an ORDER
Partial Key Discriminator within owner's scope line_number distinguishes lines within one order
Relationship Always identifying (mandatory) ORDER contains ORDER_LINE
Physical Table for Weak Entity 
-- ORDER (Strong Entity)
CREATE TABLE Order (
    order_id INT PRIMARY KEY,
    order_date DATE,
    customer_id INT
);

-- ORDER_LINE (Weak Entity)
CREATE TABLE OrderLine (
    order_id INT,
    line_number INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, line_number),  -- Composite key
    FOREIGN KEY (order_id) REFERENCES Order(order_id)
);

✅ 8-Step Transformation Checklist

  1. 1
    Map Regular (Strong) Entities
    Create a table for each strong entity. Include all simple attributes. Primary key becomes the table's PK.
  2. 2
    Map Weak Entities
    Create table with composite PK = Owner's PK + Partial key. Add FK to owner.
  3. 3
    Handle 1:1 Relationships
    Merge into single table, or add PK of one side as FK to the other. Put FK on optional side if applicable.
  4. 4
    Handle 1:N Relationships
    Add PK of the '1' side as FK on the 'N' side. Include any relationship attributes with the FK.
  5. 5
    Handle M:N Relationships
    Create junction/associative table with PKs of both entities as composite PK. Include relationship attributes.
  6. 6
    Resolve Multi-Valued Attributes
    Create separate table with FK to original entity. Composite PK = FK + multi-valued attribute.
  7. 7
    Handle N-ary Relationships
    Create associative entity with PKs of all participating entities. Composite PK = all FKs combined.
  8. 8
    Transform Superclass Structures
    Choose strategy: Roll-Up, Push-Down, or Relate. Consider exclusive vs overlapping subtypes.

📊 Subclass Transformation Strategies

Strategy Structure Best For Pros Cons
Roll-Up Single table with discriminator Few differences between subtypes No joins needed NULL values, large table
Push-Down Separate table per subtype Many differences, query by subtype No NULLs, clean tables Duplicate PK columns
Relate Superclass table + subtype tables Shared + unique attributes Clean inheritance model Requires joins
Roll-Up + Flags Single table with boolean columns Overlapping subtypes Handles overlap Many flag columns
Subclass Strategy Examples 
-- ROLL-UP: Single table for Person, Student, Employee
CREATE TABLE Person (
    person_id INT PRIMARY KEY,
    name VARCHAR(100),
    address VARCHAR(200),
    person_type CHAR(1),  -- 'S' for Student, 'E' for Employee
    -- Student-specific (NULL for non-students)
    student_id VARCHAR(20),
    major VARCHAR(50),
    gpa DECIMAL(3,2),
    -- Employee-specific (NULL for non-employees)
    employee_id VARCHAR(20),
    department VARCHAR(50),
    salary DECIMAL(10,2)
);

-- PUSH-DOWN: Separate tables
CREATE TABLE Person (
    person_id INT PRIMARY KEY,
    name VARCHAR(100),
    address VARCHAR(200)
);

CREATE TABLE Student (
    person_id INT PRIMARY KEY,
    student_id VARCHAR(20),
    major VARCHAR(50),
    gpa DECIMAL(3,2),
    FOREIGN KEY (person_id) REFERENCES Person(person_id)
);

CREATE TABLE Employee (
    person_id INT PRIMARY KEY,
    employee_id VARCHAR(20),
    department VARCHAR(50),
    salary DECIMAL(10,2),
    FOREIGN KEY (person_id) REFERENCES Person(person_id)
);

-- RELATE: Superclass + subtype tables
CREATE TABLE Person (
    person_id INT PRIMARY KEY,
    name VARCHAR(100),
    address VARCHAR(200)
);

CREATE TABLE Student (
    person_id INT PRIMARY KEY,
