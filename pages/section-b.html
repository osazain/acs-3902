Section B: EERD (Enhanced Entity Relationship Diagrams)

Questions: 40+ Points: 1000

Study Mode: Learn EERD Concepts

Review notation, concepts, and examples before taking the quiz.

🏛️ Chen Notation

Conceptual/Logical

Basic Symbols

  • Rectangle - Entity
  • Diamond - Relationship
  • Oval - Attribute
  • Double Oval - Multi-valued Attribute

Advanced Symbols

  • Double Rectangle - Weak Entity
  • Double Diamond - Identifying Relationship
  • Circle with 'd' - Exclusive Subtypes
  • Circle with 'o' - Overlapping Subtypes

Cardinality

1:1 One to One
1:N One to Many
M:N Many to Many
Example: Order/Line Item
+=========+ 1 contains N +=========+ ║ ORDER ║◄══════════════◆══════════════════►║LINE ITEM║ +=========+ +=========+

Double rectangle = Weak Entity, Double diamond = Identifying Relationship

🔧 IE Notation

Industry Standard

Entity Shapes

  • Square corners - Strong Entity
  • Rounded corners - Weak Entity

Relationship Lines

  • Solid Line - Identifying Relationship
  • Dotted Line - Non-Identifying Relationship

Crow's Foot Cardinality

| One and only one
O| Zero or one
< Many
O< Zero or many

Subtypes

  • Circle with X - Exclusive Subtypes
  • Circle with O - Overlapping Subtypes
Example: Customer/Order
+-----------+ +---------+ │ CUSTOMER │───────O<─────────│ ORDER │ +-----------+ +---------+ Zero or many

A customer can have zero or many orders

🧬 Superclass/Subclass (Inheritance)

Important for Midterm

The "Is A" Test

Every subtype relationship must pass: "X IS A Y"

✓ "House IS A Dwelling"
✓ "Student IS A Person"
✗ "Active IS A Customer" Status, not a type
✗ "Premium IS A Customer" Tier/attribute, not a type

Exclusive (Disjoint) Subtypes

Entity can be ONLY ONE subtype

  • Chen: Circle with 'd'
  • IE: Circle with 'x'
Example: Dwelling = House OR Condo OR Apartment

Overlapping Subtypes

Entity can be MULTIPLE subtypes

  • Chen: Circle with 'o'
  • IE: Circle with 'o'
Example: Person = Student AND Employee

Inheritance Rules

  • Subclasses inherit ALL attributes from superclass
  • Subclasses inherit ALL relationships from superclass
  • Subclasses can have their own additional attributes
  • Subclasses can have their own additional relationships
Complete Example: Person Hierarchy
PERSON SIN, FirstName, LastName, DOB → has Identity Card
STUDENT + SIN, FirstName, LastName, DOB + StudentID, Major + gets Transit Pass
EMPLOYEE + SIN, FirstName, LastName, DOB + EmployeeID, Department

🔗 Weak Entities & Identifying Relationships

High Importance

What is a Weak Entity?

  • Cannot be uniquely identified by its own attributes alone
  • Depends on another entity (the owner/identifying entity)
  • Borrows part of its primary key from the owner

Identifying Relationship

  • Child's PK includes parent's PK
  • Chen: Double diamond
  • IE: Solid line

Partial Key (Discriminator)

  • Identifies weak entity only within its owner
  • Chen: Dashed underline
  • Logical: Part of composite PK

Mandatory vs Weak

Mandatory Only:

Must have relationship but has own PK

Rental Agreement has AgreementID

Weak Entity:

Needs parent's PK for identification

Line Item needs OrderID + LineNum

Common Weak Entity Examples

Parent (Strong) Child (Weak) Partial Key Complete PK
Order Line Item LineNum (OrderID, LineNum)
Employee Dependent DependentName (EmpID, DepName)
Building Room RoomNum (BuildingID, RoomNum)
Invoice Payment PaymentNum (InvoiceID, PaymentNum)

🎭 Role Names

Clarifies Relationships

Purpose

Role names clarify the role an entity plays in a specific relationship.

Rules

  • Role names are nouns (not verbs)
  • Particularly useful in recursive relationships
  • Help when the same entity has multiple relationships

Examples

Recursive (Employee manages Employee):
manager → supervises → subordinate
Kijiji (User in Order):
buyer ← places → seller
University (Person in Course):
instructor → teaches → student

Key Concepts Summary

Weak Entity Pattern

  1. Double rectangle (Chen) or rounded corners (IE)
  2. Double diamond (Chen) or solid line (IE)
  3. Partial key with dashed underline (Chen)
  4. Complete PK = Parent PK + Partial Key

"Is A" Test

For valid subtype:

  1. Must pass "X IS A Y" semantically
  2. Subtypes should have different attributes
  3. Subtypes should have different relationships
  4. Statuses (Active, Premium) ≠ Subtypes

Inheritance

Subclasses inherit:

  • ✓ All attributes from superclass
  • ✓ All relationships from superclass
  • + Can have additional attributes
  • + Can have additional relationships

Mandatory vs Weak

Mandatory: Must have relationship but has own PK

Weak: Needs parent's PK for identification

Rental Agreement: Mandatory but NOT Weak
