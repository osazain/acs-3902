Dynamic Attribute Pattern

What is the Dynamic Attribute Pattern?

A modeling pattern for relational databases that allows fully extendable data models where users can define their own attributes at runtime. This is particularly useful when different entities of the same type need different properties.

Use Case: Real Estate Application

Real estate listings need different attributes based on property type:

  • House: Number of bedrooms, bathrooms, square footage, yard size
  • Condo: Unit number, condo fees, amenities, parking
  • Commercial: Zoning type, loading docks, square footage, parking spaces
  • Land: Zoning, utilities available, access type
Implementation: Three Normalized Tables

The Dynamic Attribute Pattern uses three related tables:

TablePurpose
PropertyThe main entity being described (PropertyID, Address, Price, etc.)
Attribute DefinitionsStores attribute metadata (Name, Data Type, Description)
Attribute ValuesStores actual values (PropertyID, AttributeID, Value)
Limitation: This pattern has complexities and shortcomings in validation since attributes are stored as values rather than columns.
🔧

MS Access Implementation

The database diagram shows three related tables implementing the Dynamic Attribute Pattern:

📁 Property Table

The main entity table containing core property information that is consistent across all property types.

📋 Attribute Definitions Table

Stores attribute name, definition, and data type. Defines what attributes are available in the system.

💾 Attribute Values Table

Stores the actual values for each property's attributes. Links Property to Attribute Definitions.

Introduction to NoSQL Databases

What is NoSQL?

NoSQL (Not Only SQL) refers to alternative data representation and storage systems beyond traditional relational databases. They are designed for scalability, flexibility, and specific use cases where relational models may not be optimal.

Why NoSQL?

  • Flexible Schemas: Documents can have different fields without ALTER TABLE
  • Scalability: Designed for horizontal scaling across multiple servers
  • Performance: Optimized for specific access patterns
  • Big Data: Handle large volumes of unstructured or semi-structured data
  • Development Speed: Faster iteration for changing requirements

Types of NoSQL Databases

📄 Document Stores

MongoDB, CouchDB - Store data as JSON-like documents

🔑 Key-Value Stores

Redis, DynamoDB - Simple key-value pairs

📊 Column Family

Cassandra, HBase - Wide column stores

🔗 Graph Databases

Neo4j - Graph-based relationships

🍃

MongoDB Overview

MongoDB

A document-oriented NoSQL database that stores data in flexible, JSON-like documents called BSON (Binary JSON). It is one of the most popular NoSQL databases.

MongoDB Terminology vs Relational

MongoDB Term Relational Equivalent Description
Collection Table A group of related documents
Document Row A single record stored as BSON
Field Column A key-value pair in a document
_id Primary Key Unique identifier for each document
Embedding Denormalization Storing related data within a document
Referencing Foreign Key Linking documents using ObjectID references

What MongoDB Supports

  • Constraints: Called "validations" in MongoDB
  • Relationships: Can express relationships between entities
  • Programmatic DML: Equivalent operations to SQL INSERT, UPDATE, DELETE
  • Indexes: For improved query performance
  • Aggregation: Pipeline-based aggregation framework
Limitation: MongoDB does not have true referential integrity enforcement like foreign keys in relational databases. References are maintained at the application level.
📄

JSON and BSON

JSON

JavaScript Object Notation - Text-based data interchange format

{
  "name": "John",
  "age": 30,
  "isStudent": false
}

Human-readable format used for APIs and data exchange

BSON

Binary JSON - Binary-encoded serialization format used by MongoDB

{
  "_id": "ObjectId('...')",
  "name": "John",
  "age": 30,
  "created": "ISODate(...)"
}

More efficient storage, supports additional types like dates and binary

BSON Data Types

MongoDB Document Format

MongoDB stores data as documents within collections. Each document is a self-contained unit of data.

Example: Property Listing Document

{
  "_id": "ObjectId('507f1f77bcf86cd799439011')",
  "address": "123 Main St",
  "price": 450000,
  "type": "house",
  "bedrooms": 4,
  "bathrooms": 2.5,
  "sqft": 2200,
  "features": ["garage", "pool", "garden"],
  "agent": {
    "name": "Jane Smith",
    "phone": "555-1234",
    "email": "jane@realty.com"
  }
}
Key Features:
  • Documents are schema-flexible - no fixed structure required
  • Can embed objects directly within a document
  • Can store arrays of values
  • Each document has a unique _id field automatically generated
🧩

Complex Attributes and Objects

MongoDB supports nested objects (embedded documents) which allow you to represent complex, hierarchical data structures within a single document.

Embedding vs Referencing

✅ Embedding

Store related data within the same document

{
  "product": "Laptop",
  "manufacturer": {
    "name": "TechCorp",
    "country": "USA"
  }
}

Best for: data that is accessed together, bounded size

🔗 Referencing

Store related data in separate documents and reference by ID

{
  "product": "Laptop",
  "manufacturer_id": "ObjectId('...')"
}

Best for: many-to-many relationships, unbounded arrays

When to Embed vs Reference
Use Embedding WhenUse Referencing When
Data is accessed together frequentlyData is accessed independently
The embedded data has bounded sizeThe array could grow unbounded
You need atomic updatesYou need normalized data (reduce duplication)
Relationship is composition (part-of)Relationship is association
📊

Multi-valued Attributes (Arrays)

MongoDB documents can contain arrays as field values, allowing you to store multiple values in a single field without creating a separate collection.

Array Examples

{
  "colors": ["red", "blue", "green"],
  "scores": [95, 87, 92],
  "mixed": ["hello", 42, true],
  "objects": [
    { "type": "A", "value": 100 },
    { "type": "B", "value": 200 }
  ]
}
Querying Arrays:

MongoDB provides special operators for querying arrays:

  • $in - Match any value in array
  • $all - Match all values in array
  • $size - Match array of specific size
  • $elemMatch - Match array elements with conditions
🔑

Surrogate Keys in MongoDB

ObjectID

MongoDB automatically generates a unique identifier called ObjectID for each document. This is a 12-byte binary value that ensures uniqueness across distributed systems.

ObjectID Structure

{
  "_id": "ObjectId('507f1f77bcf86cd799439011')"
}

// Components of ObjectId:
// - 4 bytes: Timestamp (when created)
// - 5 bytes: Random value (machine ID + process ID)
// - 3 bytes: Counter (incrementing value)

Key Properties

  • Uniqueness: Guaranteed unique across all documents in all collections
  • Distributed-ready: Designed for distributed database environments
  • Sortable: ObjectIDs can be sorted by creation time
  • Flexible: You can also use your own values (strings, integers) as _id
Custom _id Example
// You can use custom IDs if needed:
{
  "_id": "PROP-001",
  "address": "123 Main St",
  "price": 450000
}
🔗

Object Referencing

MongoDB uses ObjectID references to "point" at other documents, similar to how foreign keys work in relational databases.

Example: Property with Agent Reference

// Property document references Agent
{
  "_id": "ObjectId('507f1f77bcf86cd799439011')",
  "address": "123 Main St",
  "price": 450000,
  "agent_id": "ObjectId('507f191e810c19729de860ea')"
}

// Agent document
{
  "_id": "ObjectId('507f191e810c19729de860ea')",
  "name": "Jane Smith",
  "email": "jane@realty.com"
}
Important Limitation:

MongoDB does not enforce referential integrity. If an agent is deleted, the property document will still contain a reference to a non-existent agent_id. This must be handled at the application level.

Comparison to Relational Foreign Keys

Aspect Relational (FK) MongoDB (Reference)
Integrity Enforcement Automatic by database Application responsibility
Cascade Operations Built-in (ON DELETE CASCADE) Manual implementation
Join Performance Optimized with indexes Requires $lookup aggregation
Data Consistency Strong (ACID) Eventual consistency model
📊

Summary: Relational vs NoSQL

Feature Relational (SQL) MongoDB (NoSQL)
Data Model Tables with fixed schema Collections with flexible documents
Schema Fixed, defined upfront Flexible, can vary per document
Relationships Foreign keys with constraints Embedding or referencing
Referential Integrity Enforced by database Application responsibility
Unique IDs Auto-increment integers ObjectID (globally unique)
Best For Structured data, strong consistency Flexible schemas, scaling, rapid development
