Common Table Expressions: Concept

Provides the ability to "build" temporary tables as part of a SQL query. These only exist as the statement is run.

SQL Query
WITH t1 AS (...), t2 AS (...)
↓
T1 "Temp" Table
Generated First
→
T2 "Temp" Table
Generated Next
↓
Final Query
Can refer to t1/t2

In the CTE query the "temp" tables exist only during the query execution.

📋

Non-Recursive CTE Examples

-- Non-recursive CTE
WITH high_earners AS (
    SELECT emp_id, emp_name, salary
    FROM employee
    WHERE salary > 80000
)
SELECT * FROM high_earners
ORDER BY salary DESC;

-- Multiple CTEs
WITH
    dept_avg AS (
        SELECT dept_id, AVG(salary) AS avg_sal
        FROM employee GROUP BY dept_id
    ),
    above_avg AS (
        SELECT e.*, d.avg_sal
        FROM employee e
        JOIN dept_avg d ON e.dept_id = d.dept_id
        WHERE e.salary > d.avg_sal
    )
SELECT * FROM above_avg;
📊

PostgreSQL Multi-CTE Example: Regional Sales

Example from PostgreSQL documentation – get summary information about products in the top regions. This example uses two table creations in the WITH clause.

WITH regional_sales AS (
    SELECT region, SUM(amount) AS total_sales
    FROM orders
    GROUP BY region
),
top_regions AS (
    SELECT region
    FROM regional_sales
    WHERE total_sales > (
        SELECT SUM(total_sales)/10
        FROM regional_sales
    )
)
SELECT
    region,
    product,
    SUM(quantity) AS product_units,
    SUM(amount) AS product_sales
FROM orders
WHERE region IN (SELECT region FROM top_regions)
GROUP BY region, product;

Step 1: Build Region summary (regional_sales)

Step 2: Isolate regions with sales > 1/10th of total (top_regions)

Step 3: Show product report limited to top regions

Recursive CTE: Company Org Charts

A company org chart of employees shows who manages or supervises others in the organization. Following the tree downward, some branches will have more levels than others.

In a data model, a simple recursive relation is all that is needed to create the possible need for a recursive query.

Common Table Expressions can be placed into a "recursive mode" to repeat a query a dynamic number of times.

Query Difficulty Levels

EASY Who reports to James?

MEDIUM How many people does James supervise?

HARD Who are subordinate to James in the company?

Recursive CTE for Org Charts

How many people are below James? How many descendants does James have?

WITH RECURSIVE org_hierarchy AS (
    -- Anchor member: start with CEO (no manager)
    SELECT employee_id, name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive member: find subordinates
    SELECT e.employee_id, e.name, e.manager_id, oh.level + 1
    FROM employees e
    JOIN org_hierarchy oh ON e.manager_id = oh.employee_id
)
SELECT REPEAT('  ', level) || name AS org_chart, level
FROM org_hierarchy
ORDER BY level, name;
Key Components

Anchor: The starting point (WHERE firstName = 'James')

UNION: Appends each result from each recursion to the previous one

Recursion: Continues until nothing is returned

Final Query: Operates on the final result once all recursion is done

🔍

How Does the Recursion Actually Work?

Each step indicates the CTE query being run (data set simplified to limit recursion).

1
Anchor Query

Start with James (personId = 2, managerId = null)

SELECT * FROM Person WHERE firstName = 'James'
2
First Recursion

Find direct reports of James (Peter, Andrew - managerId = 2)

JOIN all_subordinates ON p.parentId = sub.personId
3
Second Recursion

Find reports of Peter & Andrew (Judy, Mark, Linda, Mary, Lee)

4
Third Recursion

Find reports of Judy (Joe, Tyler)

✓
Recursion Ends

No more subordinates found - return final result

-- Count all subordinates of James (excluding James himself)
WITH RECURSIVE all_subordinates(personId, firstName) AS (
    SELECT personId, firstName
    FROM Person
    WHERE firstName = 'James'  -- start with James (anchor)

    UNION  -- append each result from each recursion

    SELECT p.personId, p.firstName
    FROM person AS p
    INNER JOIN all_subordinates sub ON (p.parentId = sub.personId)
)
SELECT COUNT(*) - 1  -- exclude James from the count
FROM all_subordinates;

Triggers

A trigger allows us to run code/functions when certain events (INSERT, UPDATE, DELETE) occur on a table. Triggers can be used in various applications, such as maintaining database consistency, monitoring database updates, and updating derived data automatically.

Trigger vs Constraint

Unlike constraints, triggers can work at any/all data across tables in the database.

INSERT/UPDATE/DELETE
→
BEFORE/AFTER Trigger
→
Execute Function

Trigger Types

BEFORE INSERT
Run before a new row is inserted
AFTER INSERT
Run after a new row is inserted
BEFORE UPDATE
Run before a row is updated
AFTER UPDATE
Run after a row is updated
BEFORE DELETE
Run before a row is deleted
AFTER DELETE
Run after a row is deleted

Can contain logic to accept/reject the change, or to do additional processing.

💡

Trigger Use Cases

  • Don't allow an employee to be added unless the department they are assigned to has a status of "hiring".
  • Whenever someone adds an employee, also add a record in the payroll table.
🔑

OLD and NEW Keywords

OLD
A keyword that references the current content of a row (updates, deletes). OLD.column_name refers to a column before it is deleted or updated.
NEW
A keyword to reference a value in a new row (inserts, updates). NEW.column_name refers to a column of a new row that is inserted or updated.

In UPDATE triggers, both OLD and NEW are available. In INSERT triggers, only NEW exists. In DELETE triggers, only OLD exists.

📝

Useful Application: Employee Change Log

Complete example of a trigger that logs salary changes for auditing purposes.

Step 1: Create the Log Table

-- A table for logging updates (simplified)
CREATE TABLE employee_change_log (
    employee_id VARCHAR(9) NOT NULL,
    changedAt TIMESTAMP NOT NULL,
    salary NUMERIC(13,2) NOT NULL,
    PRIMARY KEY (employee_id, changedAt)
);

Step 2: Define the Trigger Function

CREATE FUNCTION write_log()
RETURNS TRIGGER AS $example_table$
BEGIN
    INSERT INTO employee_change_log(employee_id, changedAt, salary)
    VALUES (NEW.employee_id, CURRENT_TIMESTAMP, NEW.salary);

    RETURN NEW;
END;
$example_table$ LANGUAGE plpgsql;

Step 3: Declare the Trigger

CREATE TRIGGER log_salary_change_trigger
AFTER UPDATE ON employee
FOR EACH ROW EXECUTE PROCEDURE write_log();

Assertions

Useful for specifying user-defined constraints that can span multiple tables and can be checked anytime.

Example: "In order for a section to run, it has to have met the minimum and the instructor has to have been assigned."

Stored Procedures vs Functions

Feature Stored Procedures Functions
Return Value Do not return a value Can return a value
Use Case Repetitive tasks with many sequential steps Custom calculations (like built-in DBMS functions)
Usage Called with CALL statement Called within SQL statements

Index Concept

An index is used by a DBMS to improve different aspects of performance or enforce uniqueness.

📚 Textbook Analogy

Let's think about an index in a textbook...

What if a large textbook had no index?

  • Why is an index valuable?
  • The data will be written in any order.
  • An index will be maintained as data is accumulated to help us find data faster.
How Indexes Work

Indexes are maintained as data changes to allow more efficient enforcement of unique constraints and/or allow efficient searching/retrieval.

Unique indexes are created automatically to support unique constraints (PK, AK).

Performance indexes need to be intentionally created by the database designer/administrator.

Imagine that data is written/arrives in the table "at random". Without indexes, the RDBMS would need to "scan" the whole table each time.

📊

Index Types

B-Tree Index
Default index type. Best for equality and range queries.
Hash Index
Best for equality comparisons only. Not for range queries.
Composite Index
Index on multiple columns. Order matters!
Covering Index
Index includes all columns needed for query (INCLUDE clause).
🔧

Index Creation Examples

-- Create index on a single column
CREATE INDEX idx_employee_name ON employee(last_name);

-- Composite index on multiple columns
CREATE INDEX idx_emp_dept_sal ON employee(dept_id, salary);

-- An index to help us find films by director
CREATE INDEX director_idx ON films (director);

-- A unique index to prevent duplicate users based on email
-- (When we declare a unique constraint, the database secretly creates a unique index)
CREATE UNIQUE INDEX ak_email ON user (email);

-- Covering index with INCLUDE clause
-- The INCLUDE columns are stored in the index but not part of the index key
CREATE INDEX idx_emp_covering ON employee(dept_id)
INCLUDE (emp_name, salary);
Covering Index Benefits

A covering index "covers" a query by including all columns needed. The query can be satisfied entirely from the index without accessing the table.
