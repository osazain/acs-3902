Lecture 10

Normalization

⏱️ 2 hours📖 BCNF, 1NF, 2NF, 3NF🎯 Advanced
🎯

Why Normalize?

Database Normalization Objective:

"To create relations where every dependency is on the key, the whole key, and nothing but the key."

Goals of Normalization:

  • Provide well-structured designs that encourage or enforce highly reliable data
  • Minimize redundancy that could cause inconsistencies or loss of integrity
  • A fully normalized relation is about a single concept
Redundancy Causes Problems:
  • Data inconsistency (same data stored in multiple places)
  • Update anomalies (changing data in one place but not others)
  • Insert anomalies (unable to add certain data without adding other unrelated data)
  • Delete anomalies (losing important data when deleting other data)
🔗

Functional Dependencies

Notation: A → B (A determines B)

"If you want the value for B (reliably without duplication), you need to use A to determine it."

Examples
  • StudentID → StudentName (Given a StudentID, there is exactly one StudentName)
  • ISBN → Title, Author, Publisher (Given an ISBN, you can determine all book details)
  • CourseCode → CourseName, Credits

Key Terms:

Determinant

The attribute (A) on the left side of the arrow that determines another attribute.

Dependent

The attribute (B) on the right side of the arrow that is determined by A.

1️⃣

First Normal Form (1NF)

The "Has One" Test

A relation is in 1NF if it passes three rules:

Rule 1: Atomic Attributes

All attributes stored in the relation are judged to be atomic (cannot be broken down further).

Rule 2: Single-Valued Attributes

All attributes stored in the relation are single-valued.

Rule 3: Same Number of Attributes

All records have the same number of attributes.

Violations of 1NF:
  • Multi-valued attributes: "Authors" containing "Smith, Jones, Lee"
  • Non-atomic composite attributes: "Address" containing "123 Main St, Toronto, ON"
  • Repeating groups: Columns like Phone1, Phone2, Phone3
🔑

Boyce-Codd Normal Form (BCNF)

Definition:

A relation is in BCNF if:

  1. It passes 1NF rules
  2. All determinants are candidate keys
  3. The attributes in the table are all "describing one thing"

The Pattern Approach to BCNF:

The "Does it have more than one attribute describing the same thing?" Test

This tests for embedded entities - a determinant that is not a candidate key.

  • Mileage_KM and Mileage_Mi both describe the same concept → REDUNDANT
  • Make_Abbr describes Make → EMBEDDED ENTITY
💔

Example: Dating Profile Table (NOT Normalized)

ID Nickname Email DOB Prov Province Interests
1 Josey jmall@gmail.com 2002-12-14 MB Manitoba Star Trek, Gaming
2 Mellow_Guy jkr@Hotmail.com 2005-04-23 SK Saskatchewan Cool_J
3 ll@protonmail.com 1998-08-20 AB Alberta Knitting, Walks, Classical Music
4 Josey hello@yahoo.ca 1995-05-06 AB Alberta Cars, Motorcycles, Trucks
Problems with this table:
  • Multi-valued Interests field: "Star Trek, Gaming" and "Knitting, Walks, Classical Music" contain multiple values
  • Province name redundancy: "Manitoba", "Saskatchewan", "Alberta" are stored repeatedly with province codes
  • Nickname uniqueness issue: "Josey" appears twice with different IDs
  • Data inconsistency: Same province could be written differently

Normalization Solution:

1 Profile Table (Core Profile Info) 
CREATE TABLE Profile (
    ProfileID INT PRIMARY KEY,
    Nickname VARCHAR(50),
    Email VARCHAR(100),
    DOB DATE,
    ProvinceCode CHAR(2),
    FOREIGN KEY (ProvinceCode) REFERENCES Province(ProvinceCode)
);
2 Province Table (Lookup) 
CREATE TABLE Province (
    ProvinceCode CHAR(2) PRIMARY KEY,
    ProvinceName VARCHAR(50)
);
3 Interest Table (Multi-valued Attribute) 
CREATE TABLE Profile_Interest (
    ProfileID INT,
    Interest VARCHAR(50),
    PRIMARY KEY (ProfileID, Interest),
    FOREIGN KEY (ProfileID) REFERENCES Profile(ProfileID)
);
📚

Example: Book Table

NOT in 1NF (Before):

Book Table (Problems)
  • Multi-valued "Authors" field: "John Smith, Jane Doe"
  • Non-atomic "Book Details" field: Contains multiple pieces of information

Normalized to 1NF (After):

1 Book Table 
CREATE TABLE Book (
    ISBN VARCHAR(13) PRIMARY KEY,
    Title VARCHAR(200),
    Format VARCHAR(20),
    Category VARCHAR(50)
);
2 Author Table 
CREATE TABLE Author (
    AuthorID INT PRIMARY KEY,
    AuthorName VARCHAR(100)
);

CREATE TABLE Book_Author (
    ISBN VARCHAR(13),
    AuthorID INT,
    PRIMARY KEY (ISBN, AuthorID),
    FOREIGN KEY (ISBN) REFERENCES Book(ISBN),
    FOREIGN KEY (AuthorID) REFERENCES Author(AuthorID)
);
🚗

Example: Vehicle Table (NOT Properly Normalized)

Vehicle Table (Problems)
VehicleID Make Make_Abbr Model Year Mileage_KM Mileage_Mi
1 Toyota TOY Camry 2020 50000 31068
Problems:
  • Mileage_KM and Mileage_Mi: Both describe the same concept (mileage) → REDUNDANT
  • Make_Abbr describes Make: "TOY" is an embedded entity that describes "Toyota"

BCNF Normalization Solution:

1 Make Table (Embedded Entity) 
CREATE TABLE Make (
    MakeID INT PRIMARY KEY,
    MakeName VARCHAR(50),
    MakeAbbr CHAR(3)
);
2 Vehicle Table (Single concept, stores one unit) 
CREATE TABLE Vehicle (
    VehicleID INT PRIMARY KEY,
    MakeID INT,
    Model VARCHAR(50),
    Year INT,
    Mileage INT,  -- Only one mileage field!
    FOREIGN KEY (MakeID) REFERENCES Make(MakeID)
);
📊

Normalization Summary

Normal Form Key Requirement Problem Solved
1NF Atomic, single-valued attributes; same number of attributes Multi-valued and composite attributes
BCNF All determinants are candidate keys; one concept per table Embedded entities and redundant data
Pattern Approach to BCNF:

Look for: "Does the table have more than one attribute describing the same thing?"

  • Finding an embedded entity (a determinant that is not a candidate key)
  • Split into separate tables to eliminate redundancy
