SQL Command Categories

SQL commands are organized into four main categories:

DDL
Data Definition Language
CREATE, ALTER, DROP

Full Commands:

  • CREATE DATABASE, CREATE SCHEMA
  • CREATE DOMAIN, CREATE TABLE
  • CREATE VIEW
  • ALTER TABLE
  • DROP statements
DML
Data Manipulation Language
SELECT, INSERT, UPDATE, DELETE

Purpose:

Create, Modify, Delete, and Select/Analyze Data

DCL
Data Control Language
GRANT, REVOKE

Full Commands:

  • CREATE USER
  • CREATE ROLE
  • GRANT
  • REVOKE
TCL
Transaction Control Language
COMMIT, ROLLBACK, SAVEPOINT

Full Commands:

  • BEGIN/END TRANSACTION
  • COMMIT
  • ROLLBACK
  • SAVEPOINT

Data Control Language (DCL) Concepts

All about security and privileges. Defines users, roles, and controls what they can see and do.

User
→
may be assigned
→
Role
→
is granted
→
Privilege
Privileges: SELECT, INSERT, UPDATE, DELETE for a Structural Object (table, schema, view, etc.)
DCL Example (PostgreSQL)
-- Create role and user
CREATE ROLE instructor;
CREATE USER sousa;

-- Assign role to user
GRANT instructor TO sousa;

-- Grant privileges to role
GRANT SELECT ON course TO instructor;
GRANT ALL PRIVILEGES ON course TO registrar;

-- Meaning: "Instructors can read data from the course table."
--          "Registrars can do anything to the course table."
Best Practice

Users can be granted privileges directly, but in practice, a role is preferred for managing access. Roles make it easier to manage permissions for groups of users.

Query Processing Order (7 Steps)

The database optimizer determines the exact nature of how and when rows are retrieved and processed. This is the conceptual execution sequence (different from the writing order!):

  1. Rows Retrieved - Rows are retrieved from the referenced tables (cartesian product)
  2. WHERE Applied - Rows that match the WHERE criteria are retained
  3. JOIN Applied - Rows are joined based on join criteria
  4. Aggregation Tabulated - [if aggregating] Aggregation is tabulated and organized into non-overlapping groups according to values of the grouping attributes
  5. HAVING Applied - [if aggregating] Groups that match the HAVING criteria are retained
  6. ORDER BY Applied - The rows (or groups) remaining are presented to the user in sequence according to the ORDER BY attribute values
  7. LIMIT Applied - If LIMIT is used, the final rows shown are limited 
-- Processing Order (different from writing order!)
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

INNER JOIN Returns only matching rows from both tables Intersection

FULL OUTER JOIN All rows from both tables Union

Theta Join vs ANSI Join

Old-Style Theta Join (NOT Preferred)

The join is specified as a WHERE condition. This is known as a "theta join" or "theta style" join.

-- Theta join: Join condition in WHERE clause
SELECT Fname, Lname, Address
FROM EMPLOYEE, DEPARTMENT
WHERE Dname='Research' AND Dnumber = Dno;
ANSI JOIN Syntax (Preferred)

Same query but using an ANSI standard join condition within the FROM clause.

-- ANSI join: Join condition in FROM clause
SELECT Fname, Lname, Address
FROM EMPLOYEE
JOIN DEPARTMENT ON (Dnumber = Dno)
WHERE Dname = 'Research';
Why ANSI Joins are Preferred
  • Clearer syntax - the logic to join tables is separate and distinct from constraining criteria
  • Easier to write and manage as queries become complex
  • Better readability for multi-table joins
  • Standard across different database systems

Multi-Table Join Example

Problem: For every project located in 'Stafford', list the project number, the controlling department number, and the department manager's last name, address, and birth date.

Table Aliases

Using Table Aliases

Table aliases simplify syntax when fully qualifying column names in queries with common column names across tables.

-- Good aliases: meaningful and clear
SELECT Fname, e.Name, Address
FROM EMPLOYEE AS e
JOIN DEPARTMENT AS d ON (d.Dnumber = e.Dnumber)
WHERE d.Name = 'Research';
⚠️ Warning: Don't Use Meaningless Aliases

Using non-sensical aliases like t1, t2 can lead to errors and mistakes as it becomes easy to lose track of what the alias represents.

-- BAD: Meaningless aliases - hard to track!
SELECT Fname, t1.Name, Address
FROM EMPLOYEE AS t1
JOIN DEPARTMENT AS t2 ON (t1.Dnumber = t2.Dnumber)
WHERE t2.Name = 'Research';
📝

Column Renaming (Table-Level)

The AS keyword can also supply aliases for all columns in a table. This is like "renaming" all the columns in a table for this specific query.

SELECT Fn, e.Name, Addr
FROM EMPLOYEE AS e (Fn, Mi, Ln, Ssn, Bd, Addr, Sex, Sal, Sssn, Dno)
JOIN DEPARTMENT AS d ON (d.Dnumber = e.Dno)
WHERE d.Name = 'Research';
Note

There are times where this technique may be useful, but it is more common to simply alias the items that appear in the SELECT clause using column AS alias syntax.

✳️

Qualified Asterisk

The asterisk (*) can be used with a fully qualified table or alias to narrow its scope to columns from a specific table.

SELECT * (All Columns)
-- Returns all attributes from BOTH tables
SELECT *
FROM EMPLOYEE AS e
JOIN DEPARTMENT AS d ON (d.Dnumber = e.Dnumber)
WHERE d.Name = 'Research';
Qualified Asterisk (Specific Table)
-- Returns department name + ALL attributes from EMPLOYEE only
SELECT d.Name, e.*
FROM EMPLOYEE AS e
JOIN DEPARTMENT AS d ON (d.Dnumber = e.Dnumber)
WHERE d.Name = 'Research';

Wildcards, Pattern Matching & ILIKE

LIKE Pattern Matching
-- % matches zero or more characters
SELECT * FROM employee WHERE emp_name LIKE 'S%';  -- Starts with S
SELECT * FROM employee WHERE emp_name LIKE '%n';  -- Ends with n
SELECT * FROM employee WHERE emp_name LIKE '%son%';  -- Contains 'son'

-- _ matches exactly one character
SELECT * FROM employee WHERE emp_name LIKE '_mith';  -- 5 letters ending in 'mith'
ILIKE - Case-Insensitive LIKE (PostgreSQL)

PostgreSQL includes a special variant of the LIKE operator called ILIKE (case-insensitive like). This is unique to PostgreSQL (other RDBMS may have something similar).

Expression Result Notes
'abc' LIKE 'ABC' false Case sensitive
'abc' ILIKE 'ABC' true Case insensitive
'ABC' ILIKE 'a%' true Case insensitive with wildcard
📊

BETWEEN Operator

Evaluates if an expression or attribute falls within a range. This is equivalent to using two expressions with >= and <= operators.

value BETWEEN low AND high;
Important: BETWEEN is Inclusive

The BETWEEN operator is inclusive, meaning it includes the upper and lower boundary values.

-- This evaluates to TRUE (29 is included!)
29 BETWEEN 21 AND 29  -- Result: TRUE

-- Equivalent to:
29 >= 21 AND 29 <= 29  -- Result: TRUE
🔄

NULL in Boolean Comparisons

Occasionally, a comparison may include a NULL value being evaluated with non-NULL values. When NULL is compared to a non-NULL value, special rules apply.

Example Scenario

Suppose age is NULL and eyes = 'blue'. How do these expressions evaluate?

eyes = 'blue' AND age > 21    -- Evaluates as: NULL
eyes = 'blue' OR age > 21     -- Evaluates as: TRUE

Truth Tables (PostgreSQL)

AND Truth Table

Practice Queries (Henry Books Database)

Example 1: Find employees in Accounting department
SELECT e.emp_name, e.salary
FROM employee e
JOIN department d ON e.dept_id = d.dept_id
WHERE d.dept_name = 'Accounting'
ORDER BY e.salary DESC;
Example 2: Find employees without managers
SELECT DISTINCT dept_id
FROM employee
WHERE manager_id IS NULL;
Example 3: Top 5 projects by budget
SELECT proj_name, budget
FROM project
WHERE budget BETWEEN 10000 AND 50000
ORDER BY budget DESC
LIMIT 5;
