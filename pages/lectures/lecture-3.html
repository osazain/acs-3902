Joins, Aggregates, Subqueries

⏱️ 2.5 hours📖 18 topics🎯 Intermediate
🔄

Visual Join Types

Understanding how different JOIN operations combine data:

INNER JOIN
Only matching rows
LEFT JOIN
All A + matching B
RIGHT JOIN
Matching A + all B
FULL OUTER JOIN
All rows from both
📚 Join Classifications

Primary Join Types: INNER, NATURAL, OUTER (LEFT/RIGHT/FULL), CROSS JOIN
Join Conditions: Equi-join (= operator) vs Non-equi-join (>, <, != operators)

🔀

CROSS JOIN (Cartesian Product)

A cross join produces a Cartesian product - every row in one table matches with every row in the other table.

-- Implicit cross join (comma syntax - omits the join command)
SELECT Publisher_Name, Book_Title
FROM Publisher, Book;

-- Explicit CROSS JOIN declaration
SELECT Publisher_Name, Book_Title
FROM Publisher CROSS JOIN Book;
💡 When to Use CROSS JOIN
  • Generating all possible combinations (e.g., pairing every student with every course)
  • Creating test data by combining datasets
  • Building matrices or pivot tables

Warning: CROSS JOIN can produce massive result sets (M × N rows). Use with caution!

🌿

NATURAL JOIN

A NATURAL JOIN automatically joins tables based on commonly named attributes (columns with the same name in both tables).

-- NATURAL JOIN - automatically joins on columns with matching names
SELECT publisher_name, book_title
FROM Publisher NATURAL JOIN Book;

-- This is equivalent to:
SELECT Publisher_Name, Book_Title
FROM Publisher
JOIN Book ON (Publisher.publisher_code = Book.publisher_code);
⚠️ NATURAL JOIN is "DUMB"

NATURAL JOIN will join on ALL columns that share the same name, even if they shouldn't be joined! This can produce unexpected results if tables have incidental column name matches.

NATURAL with Outer Joins

Because NATURAL just assumes the join condition, it can be combined with LEFT/RIGHT/FULL outer joins:

SELECT Publisher_Name, Book_Title
FROM Publisher
NATURAL LEFT OUTER JOIN Book;

SELECT Publisher_Name, Book_Title
FROM Publisher
NATURAL RIGHT OUTER JOIN Book;

SELECT Publisher_Name, Book_Title
FROM Publisher
NATURAL FULL OUTER JOIN Book;
🔄

Self-Join: Aliasing Tables

When you alias a table, the SQL statement treats it as if it were another physical table. This enables querying recursive relationships.

Self-Join for Recursive Relationships

Common example: Employee/Manager relationships where an employee has a manager who is also an employee.

-- Find employees and their managers using self-join
SELECT 
    e.emp_name AS employee,
    m.emp_name AS manager
FROM Employee e
LEFT JOIN Employee m ON (e.manager_id = m.emp_id);

Multiple Relations Between Same Tables

When a table has multiple foreign keys to the same referenced table, use aliasing to distinguish them:

-- Student has both citizenship and residency country codes
SELECT 
    s.student_number,
    c_country.country_name AS country_of_citizenship,
    r_country.country_name AS country_of_residency
FROM Student s
JOIN Country c_country ON (s.citizenship = c_country.country_code)
JOIN Country r_country ON (s.residency = r_country.country_code);

The query "believes" there are two country tables because we have aliased and joined each differently.

⚖️

Non-Equal Joins (Theta Joins)

Join conditions can use operators other than equals (=).

📊 Join Type Comparison
  • Equi-join: Join condition uses the = operator
  • Non-equi-join (Theta Join): Join condition uses any operator other than = (>, <, !=, >=, <=)

Example: Find Books Priced Higher Than a Reference

-- List all books that are more expensive than book number '0378'
SELECT b.book_code, b.book_title, b.bookprice
FROM Book b
JOIN Book price_reference ON (b.bookprice > price_reference.bookprice)
WHERE price_reference.book_code = '0378';

Aggregate Functions

COUNT(*)
Count all rows
COUNT(col)
Count non-NULL
SUM()
Sum of values
AVG()
Average value
MAX()
Maximum value
MIN()
Minimum value
-- Aggregate examples
SELECT 
    COUNT(*) AS total_employees,
    AVG(salary) AS avg_salary,
    MAX(salary) AS max_salary,
    MIN(hire_date) AS earliest_hire
FROM employee;

NULL Handling in Aggregates

⚠️ Important NULL Behaviors
  • COUNT(*) vs COUNT(column): COUNT(*) counts all rows regardless of NULLs. COUNT(column) only counts non-NULL values in that column.
  • NULLs are ignored in AVG: NULL values are discarded, NOT treated as zero. This prevents skewing the average.
  • If all values are NULL: The result is NULL (not 0), except for COUNT(*) which returns the row count.

Grade Calculation Example

-- Nulls are ignored in aggregation - they are NOT treated as zero
-- Example: Assignment scores where AS3 is not yet due (NULL)
-- AS1: 0.98, AS2: 0.90, AS3: NULL

SELECT AVG(score) AS average_score
FROM assignments;
-- Result: 0.94 (only two scores averaged)
-- If NULL was treated as 0: would be 0.62 (incorrect!)
-- Demonstrating COUNT(*) vs COUNT(column)
SELECT COUNT(*) FROM book;
-- Returns: 34 (total rows)

SELECT COUNT(publisher_code) FROM book;
-- Returns: 32 (only non-NULL publisher codes)
-- 2 books have NULL publisher_code values
🔀

GROUP BY and HAVING

-- GROUP BY aggregates rows into groups
SELECT dept_id, COUNT(*) AS emp_count, AVG(salary) AS avg_salary
FROM employee
GROUP BY dept_id;

-- HAVING filters groups (like WHERE filters rows)
SELECT dept_id, COUNT(*) AS emp_count
FROM employee
GROUP BY dept_id
HAVING COUNT(*) > 5;

-- Aggregate in HAVING without showing in SELECT
SELECT book_type, COUNT(*) AS book_count
FROM BOOK
GROUP BY book_type
HAVING AVG(bookprice) >= 50;
📝 Key Rules
  • GROUP BY Rule: Any column in SELECT that isn't in an aggregate function must be in GROUP BY.
  • NULLs in GROUP BY: NULLs are NOT ignored - groups are created to represent NULL values.
  • WHERE vs HAVING: WHERE filters rows before grouping, HAVING filters groups after aggregation.

Subquery Patterns

IN / NOT IN
Check if value exists in subquery result set
SELECT * FROM employee
WHERE dept_id IN (
    SELECT dept_id FROM department
    WHERE location = 'New York'
);
EXISTS / NOT EXISTS
Check if subquery returns any rows (often faster than IN)
SELECT * FROM employee e
WHERE EXISTS (
    SELECT 1 FROM works_on w
    WHERE w.emp_id = e.emp_id
);
Scalar Subquery
Returns exactly one value
SELECT emp_name,
    (SELECT AVG(salary) FROM employee) AS company_avg
FROM employee;
Correlated Subquery
References outer query (executes once per row)
SELECT e.emp_name
FROM employee e
WHERE e.salary > (
    SELECT AVG(salary) FROM employee
    WHERE dept_id = e.dept_id
);

Subquery in FROM Clause (Inline View)

A subquery can be used as a derived table in the FROM clause:

-- Show books where any author is 'Francis'
SELECT publisher_name, book_title, bookprice
FROM publisher
NATURAL JOIN book
JOIN (
    SELECT DISTINCT book_code
    FROM wrote w
    NATURAL JOIN author a
    WHERE a.author_last LIKE 'Francis%'
) AS francis_book ON (book.book_code = francis_book.book_code);
🔍

ANY and ALL Operators

ANY and ALL allow comparisons against a set of values returned by a subquery.

📖 Operator Definitions
  • ANY: Is the expression true for ANY of the values in the list/subquery?
  • ALL: Is the expression true for ALL of the values in the list/subquery?
-- Show books priced higher than ALL Psychology books
SELECT book_title, book_type, bookprice
FROM book
WHERE bookprice > ALL (
    SELECT bookprice
    FROM book
    WHERE book_type = 'PSY'
);

-- Show books priced higher than ANY Psychology book
SELECT book_title, book_type, bookprice
FROM book
WHERE bookprice > ANY (
    SELECT bookprice
    FROM book
    WHERE book_type = 'PSY'
);

Comparison Table: IN, ANY, and ALL

Operator Value Condition Value List Result
IN 5 IN 4, 6, 12, 18 false
ANY 10 > ANY 20, 30, 40 false
ANY 10 < ANY 20, 30, 40 true
ANY 10 = ANY 20, 30, 40 false (same as IN)
ALL 25 > ALL 12, 14, 15, 18, 3 true
ALL 25 < ALL 12, 14, 15, 18, 3 false

Note: = ANY is equivalent to IN, but != ANY is NOT equivalent to NOT IN!

DML Operations: INSERT

Inserting Complete Tuples

When providing complete attribute lists, values must match the physical order of attributes in the table.

-- Insert multiple rows with one statement
INSERT INTO EMPLOYEE
VALUES 
    ('Richard', 'K', 'Marini', '653298653', '1962-12-30', 
     '98 Oak Forest, Katy, TX', 'M', 37000, '653298653', 4),
    ('Jane', 'R', 'Thomas', '484574954', '1969-10-18', 
     '8 Sampton Drive, Lubbock, TX', 'F', 38000, NULL, 2);

Inserting Partial Tuples

Specify column names when providing partial data or using a different order:

-- Insert with specific columns
INSERT INTO EMPLOYEE (Fname, Lname)
VALUES 
    ('Richard', 'Marini'),
    ('Jane', 'Thomas');

INSERT with SELECT (Cross-Table Insert)

Use a SELECT statement to generate data to insert instead of explicit VALUES:

-- Insert data from a SELECT query
INSERT INTO WORKS_ON_INFO (Emp_name, Proj_name, Hours_per_week)
SELECT E.Lname, P.Pname, W.Hours
FROM PROJECT
NATURAL JOIN WORKS_ON
NATURAL JOIN EMPLOYEE;

-- Insert high earners into a separate table
INSERT INTO high_earners (emp_id, emp_name)
SELECT emp_id, emp_name FROM employee
WHERE salary > 80000;

DML Operations: DELETE and TRUNCATE

DELETE Command

Removes tuples from a relation. Always use WHERE to isolate rows to delete.

-- Delete specific rows
DELETE FROM EMPLOYEE
WHERE Lname = 'Brown';

-- Delete all rows (DANGER!)
DELETE FROM EMPLOYEE;

TRUNCATE Command

A more efficient command to empty an entire table. Faster than DELETE without WHERE because it doesn't log individual row deletions.

-- Efficiently remove all rows from a table
TRUNCATE TABLE EMPLOYEE;
Feature DELETE TRUNCATE
Can use WHERE clause ✅ Yes ❌ No
Speed Slower (row-by-row logging) Faster (deallocates pages)
Can be rolled back ✅ Yes (in transaction) ⚠️ Database-dependent
Resets identity columns ❌ No ✅ Yes
Triggers fire ✅ Yes ❌ No
✏️

DML Operations: UPDATE

UPDATE Multiple Columns

The SET clause determines which attributes are updated. Multiple columns can be updated in one statement.

-- Update multiple columns at once
UPDATE BOOK
SET 
    book_price = book_price * 1.25,
    book_type = 'PSY',
    publisher = 'WA'
WHERE book_code = '0378';

UPDATE with JOIN (Cross-Table Updates)

A FROM clause can be added to join other tables, using data from other relations to inform the update.

-- Increase book prices based on state inflation rates
UPDATE book
SET book_price = book_price * s.inflation_rate
FROM publisher p, state_inflation s
WHERE p.publisher_state IN ('NY', 'CA', 'TX')
AND p.publisher_state = s.state_code;  -- theta join

Note: Syntax varies by database. PostgreSQL supports theta joins in UPDATE, while other RDBMS may support both ANSI and theta joins.

🏗️

DDL Operations: DROP and ALTER

DROP TABLE with CASCADE vs RESTRICT

-- CASCADE: Drops dependent objects (constraints) to allow table removal
DROP TABLE book CASCADE;

-- RESTRICT: Only drops if no dependencies exist (usually the default)
DROP TABLE book RESTRICT;
📋 CASCADE Behavior

CASCADE drops constraints that reference the table (e.g., FK constraints on child tables). Note: The child table itself is NOT dropped - only the FK constraint is removed.

ALTER TABLE Operations

Used to add, remove, and modify columns and constraints from tables.

-- Add a column
ALTER TABLE EMPLOYEE ADD COLUMN Job VARCHAR(12);

-- Drop a column
ALTER TABLE EMPLOYEE DROP COLUMN Job CASCADE;

-- Modify column type
ALTER TABLE EMPLOYEE ALTER COLUMN Salary DECIMAL(10,1);

-- Drop and add constraints
ALTER TABLE PRODUCT DROP CONSTRAINT FK123;

ALTER TABLE product ADD CONSTRAINT fk123 
    FOREIGN KEY(cat_id) REFERENCES category (id)
    ON DELETE SET NULL
    ON UPDATE CASCADE;
🔗 Referential Integrity Actions

ON DELETE SET NULL: Sets FK to NULL when parent is deleted
ON UPDATE CASCADE: Updates FK value when parent PK changes
ON DELETE CASCADE: Deletes child rows when parent is deleted

⚠️

Common Mistakes

🚨 Watch Out For These!
  • COUNT(column) vs COUNT(*) - COUNT(*) counts all rows, COUNT(col) excludes NULLs
  • NULL in AVG: NULLs are ignored, not treated as zero - this affects your average calculation!
  • WHERE vs HAVING - WHERE filters rows before grouping, HAVING filters groups after
  • Correlated subquery performance - Executes once per outer row; can be slow on large tables
  • DELETE without WHERE - Deletes ALL rows from the table!
  • TRUNCATE cannot be filtered - TRUNCATE removes ALL rows; you cannot use WHERE
  • IN with NULL - IN with NULL in subquery returns unexpected results; use EXISTS instead
  • = ANY vs IN - = ANY is equivalent to IN, but != ANY is NOT equivalent to NOT IN
  • NATURAL JOIN danger - Joins on ALL columns with matching names, even if unintended
