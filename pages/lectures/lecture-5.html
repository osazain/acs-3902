Transactions & UNION

⏱️ 1.25 hours📖 7 topics🎯 Intermediate
💳

Transactions in Real Life

Think of examples in which a transaction involves multiple changes to data, where each involves the exchange of money electronically:

💳 Example 1: Buying something with your Debit Card

When you make a purchase with your debit card, multiple steps must all succeed:

1Verify account has sufficient funds
↓
2Deduct amount from your account
↓
3Add amount to merchants account
↓
✓Transaction complete - commit changes
🌐 Example 2: Purchasing something Online with a Credit Card

Online purchases also require multiple steps to complete successfully:

1Authorize payment with credit card company
↓
2Reserve funds on credit limit
↓
3Update order status to confirmed
↓
4Reduce inventory count
⚡ The "All or Nothing" Principle

In order for these kinds of purchases to be complete, the send/receive of the monetary amounts must both be successful. The whole thing must be complete for any of it to be completed.

💥 Failure Scenarios

Now think of a failure scenario in which there is a disruption. A dramatic one like a power failure or network failure is an example we can easily imagine:

  • What if money is deducted from your account but never reaches the merchant?
  • What if the power goes out after step 1 but before step 3?
  • What if the network connection drops mid-transaction?

Without transactions: Your data could be left in an inconsistent state.

With transactions: If any step fails, the entire transaction is rolled back - no partial updates!

📝

Transaction Syntax

A transaction can be declared or "opened" when we need the "all or none" capabilities from our database.

-- Begin a transaction
BEGIN TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE account_id = 'customer';

UPDATE accounts SET balance = balance + 100 WHERE account_id = 'merchant';

-- Commit if all statements succeed
COMMIT;
🔄 How It Works

If statement 1 fails or something else happens before the COMMIT statement is reached, none of the statements are applied. The database remains in its original state.

ROLLBACK Syntax

When working with interactive SQL, you can also issue a ROLLBACK statement when you detect something is not as it should be:

BEGIN TRANSACTION;

UPDATE student SET first_name = proper(first_name);

UPDATE student SET last_name = proper(first_name);

-- "OOPS, did I just update the last_name = first_name?"

ROLLBACK;  -- undo back to the start of the transaction
           -- as if none of the statements had been done
🔄

ACID Properties

ACID properties ensure reliable processing of database transactions. These four properties guarantee that database transactions are processed reliably:

A
Atomicity
All operations complete successfully, or none do. "All or nothing" principle - transactions are indivisible units of work.
C
Consistency
Database remains in a valid state before and after the transaction. All rules and constraints are maintained.
I
Isolation
Concurrent transactions do not interfere with each other. Each transaction executes as if it is the only one running.
D
Durability
Once committed, changes persist even if the system fails. Committed data is stored permanently.
💾

Advanced Transaction Control

-- Begin a transaction
BEGIN;

-- Perform operations
UPDATE accounts SET balance = balance - 5000 WHERE account_id = 'A';
UPDATE accounts SET balance = balance + 5000 WHERE account_id = 'B';

-- Save intermediate state
SAVEPOINT before_credit;

-- More operations
UPDATE accounts SET balance = balance + 5000 WHERE account_id = 'C';

-- Rollback to savepoint if needed
ROLLBACK TO before_credit;

-- Commit all changes
COMMIT;

-- Or rollback everything
-- ROLLBACK;

UNION vs UNION ALL

A UNION operator allows us to have the database perform set operations on two or more SQL results.

Set A
+
Set B
=
UNION
(no duplicates)
-- UNION removes duplicates
SELECT name, email FROM employees
UNION
SELECT name, email FROM contractors;

-- UNION ALL keeps duplicates (faster)
SELECT name, email FROM employees
UNION ALL
SELECT name, email FROM contractors;
⚠️

UNION Column Compatibility

For UNION to work, the queries being unioned have to provide a consistent output (number of columns, data types). The column titles for the final data set are taken from the first query.

⚠️ Warning: Data Type Compatibility Issue

Although GPA and Age might be compatible (both numeric), this final result would not make a lot of sense as the column title would be "Age":

Employees
EmpID: 100
Last Name: Toledo
DOB: 1994-08-09
Age: 25
UNION
Students
Student ID: 38474
Last Name: Solanche
DOB: 1997-08-09
GPA: 3.5
❌ Mismatch Error

A mismatch in structure or datatype will cause the UNION to error out:

Employees (3 columns)
EmpID: 100
Last Name: Toledo
DOB: 1994-08-09
UNION
Students (4 columns)
Student ID: 38474
Last Name: Solanche
DOB: 1997-08-09
GPA: 2.25
⚠️ Results in an ERROR

Each UNION query must have the same number of columns with compatible data types.

SELECT name, salary FROM managers

⚠️

PostgreSQL UNION Limitations

  • 📌Column count must match: Both SELECT statements must return the same number of columns.
  • 📌Compatible data types: Corresponding columns must have compatible types.
  • 📌Single ORDER BY: Can only use ORDER BY at the end of the entire UNION statement.
  • 📌Column names from first SELECT: Result uses column names from the first query.

ORDER BY Limitation Example

PostgreSQL only allows us to use an ORDER BY clause on the last query in the union. Other RDBMS may have different or similar limitations.

-- ❌ This query will RESULT IN AN ERROR in PostgreSQL:
-- Attempting to show the most and least expensive books using ORDER BY / LIMIT

SELECT
    'most expensive' AS grouping,
    book_title,
    bookprice
FROM book
ORDER BY bookprice DESC
LIMIT 5

UNION

SELECT
    'least expensive' AS grouping,
    book_title,
    bookprice
FROM book
ORDER BY bookprice ASC
LIMIT 5;
⚠️ Results in an Error

PostgreSQL does not allow ORDER BY in unioned queries (only the last query in the union can dictate the order).

✅ Correct Approach

To properly sort a UNION result, use ORDER BY only at the very end:

SELECT name, salary, 'manager' AS role FROM managers
UNION ALL
SELECT name, salary, 'employee' AS role FROM employees
ORDER BY salary DESC;  -- ✓ ORDER BY only on the last query
