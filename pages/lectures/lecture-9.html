8-Step Transformation

⏱️ 3 hours📖 8 steps🎯 Advanced
🔄

Mapping Logical to Physical Models

What is Physical Model Transformation?

Transforming logical EER models (Chen or Information Engineering) into physical relational database models using Information Engineering notation. This is an industry-standard process for implementing databases.

Why Information Engineering for Physical Models?

  • Industry standard for physical database modeling
  • Easier to prepare DDL (CREATE TABLE statements) from
  • More complete than Database Structure Diagrams (shows AKs, data types, constraints)
  • Used almost exclusively in applied settings
⚠️

Logical-Only Structures

Cannot Exist Physically Without Transformation

These conceptual/logical structures must be transformed before they can be implemented in a physical database:

Logical Structure Physical Transformation
Many-to-Many Relationship Add junction (third) table with FKs from both entities
Relationship with Attributes (Chen) Create table to represent the relationship
Super-Class / Sub-Class Structure Restructure to 1 or more tables (multiple approaches)
Composite Attributes Split into atomic (simple) attributes
Multi-Valued Attributes Normalize to separate child table
Derived Attributes Remove from model (unless RDBMS supports computed columns)
1 Map Regular Entities to Tables
Process
  1. For each regular entity type (exclude weak entities and superclass structures), create a relation (table)
  2. Include all simple attributes (non-composite, non-multi-valued)
  3. If composite attributes exist, split them into atomic attributes
  4. Choose one of the keys as the Primary Key (PK)
Important: When an entity has multiple candidate keys, select one as the Primary Key. Others can be marked as Alternate Keys (AK) if needed for constraints.
2 Map Weak Entities
Process
  1. Create a relation (table) for each weak entity type
  2. Include all simple attributes of the weak entity
  3. Include the PK of the owner entity as a Foreign Key
  4. The weak entity's PK = Owner's PK + discriminating attribute (partial key)
-- Example: Order Line Item (weak entity)
CREATE TABLE Order_Line_Item (
    OrderID INT,           -- Part of PK, FK to ORDER
    LineNumber INT,      -- Discriminating attribute
    ProductID INT,
    Quantity INT,
    Price DECIMAL(10,2),
    PRIMARY KEY (OrderID, LineNumber),
    FOREIGN KEY (OrderID) REFERENCES ORDER(OrderID)
);
3 Handle 1:1 Relationships
Three Approaches Available!

Unlike 1:N relationships, 1:1 relationships have multiple valid approaches. The choice depends on future flexibility needs.

Approach 1: Foreign Key Approach

Choose one relation to hold a Foreign Key referencing the other.

✓ Pros: Allows for simpler change if 1:N relationship anticipated in future
✗ Cons: Requires deciding which side gets the FK; nullable if optional
Approach 2: Merged Relationship (PREFERRED)

Merge the two relations into one table.

✓ Pros: Simplest approach; no joins needed; best performance
✗ Cons: Less flexible if relationship cardinality changes to 1:N later
Recommended: Use this approach unless future flexibility for 1:N is expected!
Approach 3: Relationship Relation (Junction Table)

Treat it like an M:N relationship with a junction table.

✓ Pros: Maximum flexibility; easy to relax to M:N later
✗ Cons: More complex; requires AK constraint to enforce 1:1
AK Constraint for 1:1

To enforce 1:1 with a junction table, add an Alternate Key (AK) on one of the FKs:

Employee_Job(EmployeeID, JobNumber, StartDate)
PK: (EmployeeID, JobNumber)
AK: (JobNumber)  -- Enforces only one employee per job
4 Map 1:N Relationship Types
Approach 1: Foreign Key (Most Common)
  1. Create the child table (N-side of relationship)
  2. Include the parent's PK as Foreign Key in the child table
  3. If the relationship has attributes, push them into the child table
Approach 2: Relationship Relation (Junction Table)

Use when anticipating future cardinality changes or needing maximum flexibility. Creates a separate table with FKs to both parent and child.

Recursive Relationships: When parent and child are the same entity (e.g., Employee supervises Employee), the FK references the same table's PK.
5 Map M:N Relationship Types
Process (Junction Table Approach)
  1. Create a junction table representing the relationship
  2. Make 1:N relationships from each participating entity to the junction
  3. Include PKs of both entities as FKs in the junction table
  4. Add any relationship attributes to the junction table
  5. Declare PK as combination of FKs (plus discriminator if needed)
-- Example: Student-Course M:N becomes Enrollment junction
CREATE TABLE Enrollment (
    StudentID INT,
    CourseID INT,
    Semester VARCHAR(10),   -- Relationship attribute
    Grade CHAR(2),          -- Relationship attribute
    PRIMARY KEY (StudentID, CourseID, Semester),  -- Composite PK
    FOREIGN KEY (StudentID) REFERENCES Student(StudentID),
    FOREIGN KEY (CourseID) REFERENCES Course(CourseID)
);
6 Resolve Multi-Valued Attributes
Process
  1. Create a new child relation (table)
  2. Include the parent entity's PK as FK
  3. The child's PK = Parent's FK + discriminating attribute
  4. If multi-valued attribute is composite, include its simple (atomic) attributes
-- Example: Employee with multiple phone numbers
CREATE TABLE Employee_Phone (
    EmployeeID INT,
    PhoneType VARCHAR(20),    -- Discriminator (Home, Work, Mobile)
    PhoneNumber VARCHAR(15),
    Extension VARCHAR(5),      -- Part of composite attribute
    PRIMARY KEY (EmployeeID, PhoneType),
    FOREIGN KEY (EmployeeID) REFERENCES Employee(EmployeeID)
);
7 Map N-ary Relationship Types
What is an N-ary Relationship?

An N-ary relationship is a modeling pattern where a weak child entity has multiple identifying relationships (typically 3 or more parents). Example: An insurance policy that involves Employee, Project, and Insurance Provider.

Basic Transformation
  1. Create the child entity table
  2. Include PKs of all participating entities as FKs
  3. Declare PK as combination of all FKs (plus discriminator if exists)
  4. Include any simple attributes of the relationship
Problem with Basic Approach: A composite PK with 3+ parts can become cumbersome for relationships from this entity to other child entities. Also, may not preserve all original constraints (like 1:1 with one parent).
Practical Design: Using Independent Key + AK Constraints

Most designers prefer a suitable independent key with unique constraints to enforce original design intent:

-- Insurance_Policy with independent key
CREATE TABLE Insurance_Policy (
    PolicyID INT PRIMARY KEY,  -- Independent key (surrogate)
    EmployeeID INT,
    ProjectID INT,
    ProviderID INT,
    PolicyNumber VARCHAR(50),
    StartDate DATE,
    -- AK constraints enforce original cardinality rules
    CONSTRAINT AK_Employee_Project_Provider
        UNIQUE (EmployeeID, ProjectID, ProviderID),  -- AK1.x
    CONSTRAINT AK_Employee_Only
        UNIQUE (EmployeeID),  -- AK2.1: one policy per employee
    FOREIGN KEY (EmployeeID) REFERENCES Employee(EmployeeID),
    FOREIGN KEY (ProjectID) REFERENCES Project(ProjectID),
    FOREIGN KEY (ProviderID) REFERENCES Provider(ProviderID)
);
8 Transform Superclass Structures
Four Approaches for Superclass Transformation

The choice depends on: Total/Partial participation, Disjoint/Overlapping constraints, and data distribution considerations.

Approach 1: Roll-Up

Single table for entire hierarchy with a type discriminator attribute.

✓ Pros: Simple (one relation); easy to query all entities
✗ Cons: Many NULLs for type-specific attributes; need check constraint; storage inefficiency if data distribution is skewed
Best for: Disjoint subtypes, smaller datasets, when you frequently query across all types
Approach 2: Roll-Up with Flags (OVERLAPPING SUBTYPES)

Single table with one boolean flag per subtype instead of a single discriminator.

✓ Pros: Handles overlapping subtypes; entity can be multiple types simultaneously
✗ Cons: Many NULL values; check constraints needed; storage inefficiency
Example: Product Hierarchy with Flags
ProductIDNameIsElectronicIsClothingWarrantySize
1Laptoptruefalse2 yearsNULL
2ShirtfalsetrueNULLLarge
3Smartwatchtruetrue1 yearMedium

Notice row 3 has both flags = true (overlapping subtypes) and can have both warranty AND size.

Approach 3: Push Down

Tables for subclasses only; superclass attributes pushed down to each subclass.

✓ Pros: Optimal storage; no NULLs; natural separation of types
✗ Cons: Hard to enforce unique key across all subclasses; many relations for consolidated view; difficult to query "all entities"
Best for: Total participation, disjoint subtypes, when types are queried separately
Approach 4: Relate (Supertype + Subtype Tables)

One table for superclass + one table per subclass with FK to superclass.

✓ Pros: Optimal storage; preserves inheritance; easy to query all or specific types
✗ Cons: More joins required; difficult to enforce disjointed constraint; more tables to manage
Best for: Most flexible; works with total/partial and disjoint/overlapping
