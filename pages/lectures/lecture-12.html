Data Warehousing and Dimensional Modeling

⏱️ 3 hours📖 Data Engineering🎯 Advanced
🏢

What is a Data Warehouse?

Definition:

An integrated data solution that pulls together data from multiple systems for reporting and analysis. It addresses the problem of data being siloed in different operational systems and enables organizations to make sense of data across multiple platforms.

Topics Covered in This Lecture:

  • Data Warehousing and its business value
  • Data Warehousing vs. Operational Systems
  • Dimensional Modeling concepts
  • Normalization vs. De-normalization (OLTP vs. OLAP)
  • Dimensions and Facts in Dimensional Modeling
  • Star Schema and Snowflake Schema
  • Fact Tables and Dimension Tables
  • Data Warehouse Environment Architecture
  • ETL (Extract, Transform, Load) processes
  • Real-world case study: Sam's Hardware
⚠️

The Business Problem

Organizations typically face these challenges when trying to analyze their data:

1️⃣ Data Integration

"My data is in 5+ different systems, I need to bring it together to make sense of it."

2️⃣ Analytical Performance

"The load placed on the system if we try to report against the live operational system can interfere with it."

3️⃣ Structural Issues

"The structures (normalized) in those databases make reporting/analysis difficult."

Data Warehouse Environment Architecture

The Basic Architectural Pattern:

The data warehouse environment follows a standard pattern that moves data from source systems through processing to end-user consumption.

Component Description Examples
Data Sources Where raw data originates Web/Portal, Email, Mobile, External Data
Source Systems Operational systems where data originates ERP, CRM, POS, HR Systems
ETL Processes Extract, Transform, and Load for data movement Data integration pipelines
Data Warehouse Integrated data ready for analysis Central repository
Data Access Layer Front-end tools for consuming data Dashboards, Reports, Multi-Dim Analysis, Scorecards
⚖️

Normalization vs. De-normalization

Normalizing (OLTP Systems)
  • By splitting up the model into additional related entities, redundancy is reduced and flexibility increases
  • Preferred for online transaction processing systems
  • Performs high volumes of small create/update/delete transactions
De-Normalizing (OLAP Systems)
  • By collapsing the model into fewer entities, data redundancy increases
  • Makes reporting, analysis, and summarizing faster
  • Preferred for systems providing read-only access to data for analysis
OLTP vs OLAP Comparison
Feature OLTP OLAP
Purpose Day-to-day operations Analytical processing
Design Normalized (3NF) De-normalized (Star/Snowflake)
Operations Small, frequent reads/writes Large, complex queries
Data Current, volatile Historical, static

Dimensional Modeling

What is Dimensional Modeling?

A de-normalized modeling technique where the goal is to:

  1. Promote High Performance Analysis
  2. Simplify Usability of Data
  3. Simplify History/Change Capture

Key Components:

  • Dimensions - The descriptive context (who, what, when, where, how)
  • Facts - The measurable events at the intersection of dimensions
  • Star Schema - Fact table at center surrounded by dimension tables
  • Snowflake Schema - Star schema with normalized dimension tables
📊

Dimensions

"A Dimension is..."
  • "The what, when, where, who, and how information that is related to a measurable event."
  • "Typically the 'noun-phrases' we find in business questions."

Facts & Measures

What is a Fact?

A Fact is a single event that occurs at the intersection of one or more dimensions. It contains measurable attributes (measures).

Measure Examples:

Quantity
Revenue
Cost
Profit
% Margin
# Sales Transactions (Count)
Key Distinction:

Facts represent quantitative data (measures) while Dimensions represent qualitative descriptors (categories for grouping and filtering).

⭐

Star Schema

Star Schema Structure:

A fact table at the center surrounded by dimension tables. It is simple and de-normalized, optimized for query performance.

Characteristics of Star Schema:

  • Simple structure - Easy to understand and navigate
  • De-normalized - All dimension data in single table per dimension
  • Fast queries - Minimal joins required for common queries
  • Fact-centric - Business metrics at the center

Fact Tables and Dimension Tables

Fact Table Characteristics:

  • Contains measures (numeric, additive values)
  • Contains foreign keys to dimension tables
  • Usually large (many millions or billions of rows)
  • Immutable - New records added, rarely updated
  • Granularity represents a business event (sale, order, shipment)

Dimension Table Characteristics:

  • Contains descriptive attributes
  • Used for filtering and grouping queries
  • Often de-normalized with many columns
  • Supports slowly changing dimensions (SCD) for historical tracking
  • Contains a primary key (usually surrogate key)

ETL (Extract, Transform, Load) Processes

What is ETL?

ETL stands for Extract, Transform, Load - the process of moving data from source systems to the data warehouse.

1
EXTRACT

Pull data from source systems (databases, files, APIs)

2
TRANSFORM

Clean, validate, and convert data to target schema

3
LOAD

Insert data into the data warehouse

ETL Transformation Examples:

  • Data type conversions
  • Data cleansing (removing duplicates, handling nulls)
  • Applying business rules and calculations
  • Aggregating detailed data
  • Encrypting sensitive data
  • Splitting or merging fields
