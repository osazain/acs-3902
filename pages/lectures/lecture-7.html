Database Index Details

Automatic Index Creation

Indexes are created automatically by the RDBMS to support constraints:

  • Unique indexes are created automatically to ensure PK and AK constraints are enforced efficiently
  • In some DBMS tools, these indexes can be seen in the admin console
💡 PGAdmin4 Note

PGAdmin4 seems to hide automatically created indexes from the admin console view. You may need to use SQL queries to see them.

-- Explicit unique index creation
CREATE UNIQUE INDEX title_idx ON films (title);

-- Basic index for performance
CREATE INDEX ON films (rating);
⚠️ Index Trade-offs

Indexes speed up SELECT queries but generally decrease performance of INSERT and UPDATE commands (indexes need to be maintained, which adds processing overhead).

Compound/Composite Indexes

An index can be applied to a combination of columns if they are often used together in criteria:

-- Compound index on multiple columns
CREATE INDEX idx_student_mm ON student (major, minor);

This index speeds up situations where we are sorting or selecting based on a combination of major and minor.

INCLUDE Clause (Covering Indexes)

The normal use of an index allows the DBMS to find the tuple quickly but still requires it to "go to the table" to fetch other attributes. The INCLUDE clause adds additional data to the index to "save the trip to the table":

-- Covering index: INCLUDE adds columns to the index
-- Example: Search by category, retrieve title and rating
CREATE INDEX idx_category_title_rating ON film (category) INCLUDE (title, rating);

NULLS FIRST Syntax

Control the placement of NULL values in the index for optimized queries:

-- Place NULL rows at the top (or bottom) to improve performance
CREATE INDEX idx_book_publisher ON book (publisher_code NULLS FIRST);

Clustered Index Concepts

🎵 Music Collection Analogy

How would you arrange CDs or Records on your shelf? You would likely organize to optimize the most frequent way you search:

  • By Artist (A-Z): Quick alphabetical lookup
  • By Genre/Mood: Dance/Party, Chill/Unwind, Heavy Rock, Classical
  • By Year: Chronological organization

Physical Ordering in Databases

We can use physical ordering to arrange data similarly:

💰 Financial Transaction Example

Sales transactions are typically processed and reported monthly. Organizing by Year/Month allows a complete set of transactions that are processed together to be retrieved at once:

2025-Jan → 2025-Feb → 2025-Mar → ... → 2025-Dec → 2026-Jan

CLUSTER Command & Maintenance

In PostgreSQL, clustering is done in two steps:

-- Step 1: Create the index
CREATE INDEX idx_transaction_year ON invoice (tx_year);

-- Step 2: Cluster the table using the index
CLUSTER invoice USING idx_transaction_year;
🔄 Maintenance Required

As more data is added, the clustered index should be occasionally recreated to ensure optimal use. PostgreSQL does NOT maintain cluster order after inserts — re-run CLUSTER periodically!

Relational Algebra Fundamentals

Why Learn Relational Algebra?

  1. Helps develop an understanding of how DBMS process and optimize SQL queries
  2. The language actually precedes SQL (existed before in the field of sets/mathematics)
  3. It is the formal language of the relational model
  4. Understanding RA helps visualize query execution and optimization

Declarative vs Procedural

SQL (Declarative)Relational Algebra (Procedural)
Non-procedural — you say what you wantOperations are implemented step by step
DBMS figures out how to get itEach operation has a defined procedure
Internally, DBMS translates SQL to RAFoundation for query optimization

SELECT Operator Properties (σ)

σ<selection_condition>(R)
  • Unary: Operates on one relation
  • Commutative: The order of select conditions does not matter
  • Selectivity: We can only get equal or fewer rows relative to the original

Rename Operator (ρ) — Three Forms

The RENAME operator is denoted by ρ (rho). Useful when a query requires multiple operations or when joining a relation to itself.

Form 1: Change Both
ρS, (B1,B2,…,Bn)(R)

Change relation name to S AND column names to B1, B2, ..., Bn

Form 2: Change Relation Only
ρS(R)

Only change the relation name to S

Form 3: Change Attributes Only
ρ,(B1,B2,…,Bn)(R)

Only change attribute names to B1, B2, ..., Bn

Sequencing Operations (←)

Breaking down operations using sequencing is not mandatory but may help as complexity increases. The arrow notation reflects "storing" a result temporarily:

-- Step 1: Create temporary relation DEP5_EMPS
DEP5_EMPS ← σDno=5(EMPLOYEE)

-- Step 2: Project from the temporary relation
RESULT ← πFname,Lname,Salary(DEP5_EMPS)

Advanced Join Operators

Cartesian Product (✕)

  • Equivalent to CROSS PRODUCT in mathematics or CROSS JOIN in SQL
  • Denoted by ✕
  • Binary set operation (from set mathematics)
  • Often produces attributes with the same name — use dot notation to qualify

THETA JOIN (θ)

A general join condition where θ (theta) is one of the comparison operators:

{=, <, ≤, >, ≥, ≠}

Each condition is expressed as: Ai θ Bj

  • Ai is an attribute of relation R
  • Bj is an attribute of relation S
  • Ai and Bj must have compatible domains (data types)

EQUIJOIN

  • Join condition uses ONLY the = comparison operator
  • Looks for a match across two relations
  • Most common type of join in practice

NATURAL JOIN (*)

  • Comparison operator denoted by * in some notations
  • Absence of a join condition often means natural join is assumed
  • Performs an equijoin using all common attributes
  • Eliminates duplicate named/valued attributes from the result

Outer Join Symbols

Query Trees & Optimization

Query Tree Basics

A query tree is used by relational database systems to represent queries internally:

  • Leaf nodes: Input relations (tables)
  • Internal nodes: Relational algebra operations
  • Execution: Bottom-up — results move up the tree as input/operand
⚡ Critical Concept: Bottom-Up Execution

Query trees are executed from the bottom up. Start with the leaf nodes (tables) and work your way up through each operation to the root (final result).

Query Tree Example

π name, salary
│
▼
σ salary > 50000
│
▼
Employee

Query Optimization Techniques

1. Push-Down Optimization

Move SELECT criteria down to the relation it came from to reduce data volume early.

2. Apply More Restrictive Select First

If we can reduce the volume of tuples earlier, the query will run faster.

3. Use Joins Instead of Cross Products

Cross products are generally avoided when optimizing queries. An explicit join is best.

🔄

RA to SQL Translation Examples

Selection: σsalary > 50000(Employee)
SELECT * FROM Employee WHERE salary > 50000;
Projection: πFname, Lname, Salary(Employee)
SELECT Fname, Lname, Salary FROM Employee;
Join with Selection: πname, dept_name(σsalary > 50000(Employee ⨝ Department))
SELECT e.name, d.dept_name
FROM Employee e
JOIN Department d ON e.dept_id = d.dept_id
WHERE e.salary > 50000;

Employee/Dependent Exercise

RA expression for: "List of female employees and their dependents":

-- SQL equivalent
SELECT Fname, Lname, Dependent_name
FROM employee
CROSS JOIN dependent
WHERE sex = 'F' 
  AND employee.ssn = dependent.ssn;

Although a cross join is done, the WHERE clause constrains the results to only those where a match exists.
