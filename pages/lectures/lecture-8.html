Data Modeling: Conceptual → Logical → Physical

The Building Design Analogy

Just as building design progresses from sketches to blueprints to construction plans, database design moves through three levels of abstraction.

Level Description Focus Output
Conceptual Main ideas and structures What entities exist and how they relate High-level ERD
Logical Platform-independent detailed model Detailed needs and requirements Complete ERD with attributes
Physical Platform-specific implementation Can be used to build real database Schema with PKs, FKs, data types
📝

ERD Notations: Chen vs Information Engineering

Feature Chen Notation Information Engineering
Best For Learning core concepts Modern applied use, physical modeling
Attribute Types Rich: multi-valued, composite, derived Simple: keys only (PK, AK, FK)
Cardinality Min/max notation (1, N, M) Crow's foot notation
Modeling Levels Conceptual and Logical Conceptual, Logical, and Physical
Industry Use Academic, learning Professional, implementation

Subtype/Supertype Structures

What are Subtypes?

When entities share many attributes and relationships but have some unique characteristics, we model them using an inheritance structure (like object-oriented programming).

Real-World Example: Dwelling Insurance

Inheritance Structure

All dwellings share common attributes (address, owner, insured value), but each type has unique characteristics:

  • Apartment: Unit number, building amenities, floor level
  • Condo: Condo fees, management company, unit factor
  • House: Lot size, garage type, basement finished
Information Engineering

Circle with line underneath = Exclusive subtype

Enhanced Chen

Triangle with subset symbol ⊂

Exclusive vs Overlapping Subtypes

Type Also Called Description Example
Exclusive Disjointed Entity can be ONLY ONE subtype A dwelling is either Apartment, Condo, OR House
Overlapping Non-exclusive Entity can be MULTIPLE subtypes simultaneously A person can be both Student AND Employee

Attribute and Relationship Inheritance

Person → Student/Faculty/Staff Example

All People have: SIN, First Name, Last Name, Date of Birth

Subtypes add:

  • Student: Student ID, Program, Enrollment Date
  • Faculty: Employee ID, Department, Rank
  • Staff: Employee ID, Department, Position

Relationship Inheritance: "All people are issued an identity card, but only students are issued a transit pass."

The "Is A" Test

✓ PASS: Valid Subtype

Reading bottom-up: "A [subtype] IS A [supertype]"
Example: "An Apartment IS A Dwelling" ✓
Example: "A Student IS A Person" ✓

✗ FAIL: Invalid Subtype

These FAIL the "Is A" test and should NOT be modeled as subtypes:

  • "A Credit Card IS A Customer" ✗ (It's a payment method!)
  • "A Debit Card IS A Customer" ✗
  • "Cash IS A Customer" ✗

Unnecessary Subtype Use

⚠️ "Examples of" vs True Subtypes

This structure may pass the "Is A" test but is unnecessary:

Customer → Canadian Customer / US Customer / Mexican Customer

If the subtypes have no unique attributes or relationships, they are just "examples of" customers, not true subtypes. Use a "country" attribute instead.

Kijiji Marketplace Example

Subtype Structure for Marketplace Items

Different item types have very different attributes:

  • General Item: Title, description, price, condition
  • Vehicle: Make, model, year, mileage, VIN, transmission
  • Real Estate: Address, bedrooms, bathrooms, square footage, property type

All items share: Listing ID, Seller, Post Date, Status

Roles and Role Names

What are Roles?

A role provides a clarifying label for an entity as it participates in a particular relationship. We use this concept in everyday language!

Everyday Examples

  • An employee teaching a course plays the role of "professor" or "instructor"
  • A person taking a course for credit plays the role of "student"
  • The same person can play different roles in different contexts!

    Roles to the Rescue: Marketplace Example

    Wrong Approach vs Role-Based Model
    ❌ WRONG: Separate Entities

    Modeling Buyer and Seller as separate entities creates a cumbersome experience: "Why do I need different login IDs when buying vs selling?"

    ✓ CORRECT: Role-Based Model

    Single User entity with roles:
    • "A user may act as a buyer in relation to an order"
    • "A user may act as a seller in relation to an order"

    Recursive Relationships with Roles

    Employee Supervision Example
    Employee
    ↔ supervises
    Employee
    Role: Manager
    Role: Subordinate

    "An employee acting as a manager supervises zero or many subordinates."

    Role Naming Rules

    💡 Important Rule

    Unlike relationship verbs, roles should be NOUNS (not verbs).
    ✓ "buyer", "seller", "manager", "subordinate", "poster"
    ✗ "buys", "sells", "manages"

    Multiple Relationships Between Same Entities

    Kijiji User and Order

    The same User entity can have multiple relationships with Order:

    • As poster: User posts and pays for an ad
    • As seller: User issues a bill of sale
    • As buyer: User signs a bill of sale

Weak Entities and Identifying Relationships

What is a Weak Entity?

A weak entity describes a situation where an entity relies on an attribute from a parent for its identity. It cannot be uniquely identified without its parent entity.

Real-World Examples

Common Weak Entity Scenarios
Weak Entity Parent Entity Why It's Weak Composite Key
Order Line Item Order Line item #1 exists in many orders Order # + Line Item #
Dependent Employee "Child" exists for many employees Employee # + Dependent Name
Course Section Course Section A exists for many courses Course Code + Section

⚠️ Don't Confuse Mandatory Participation with Weakness

✓ Mandatory but NOT Weak

Car Rental Agreement must relate to a Customer, but it has its own Agreement Number (PK). It doesn't need Customer ID to be uniquely identified.

✗ Weak Entity

Order Line Item has only a line number. Without the Order ID, you can't tell which line item #1 it is.

Notation Comparison

Concept Chen Notation Information Engineering
Weak Entity Double rectangle border Rounded corners (often omitted)
Identifying Relationship Double diamond Solid line
Non-Identifying Relationship Single diamond Dotted line
Mandatory Participation Double line (child side) Part of identifying relationship
💡 Chen Pattern Recognition

In Chen notation, these three constructs typically occur together:
1. Weak entity (double rectangle)
2. Identifying relationship (double diamond)
3. Mandatory participation on the child side (double line)

Employee & Dependent Example

Complete Weak Entity Scenario

An employee can have multiple dependents (spouse, children). Each dependent is identified by combining the Employee ID with the dependent's name or sequence number. 

DEPENDENT(EmployeeID, DependentName, Relationship, DateOfBirth)
PK: (EmployeeID, DependentName)
FK: EmployeeID references EMPLOYEE

Without EmployeeID, "John - Child" could belong to any employee!

Chen Notation Reference

Attribute Types in Chen Notation

Attribute Type Notation Description Example
Simple Oval Single-valued, atomic FirstName, BirthDate
Composite Oval connected to ovals Can be broken into parts Address → (Street, City, Province, PostalCode)
Multi-Valued Double oval Can have multiple values PhoneNumbers, EmailAddresses
Derived Dashed oval Can be calculated from other attributes Age (from BirthDate), TotalOrderAmount
Key Underlined name Primary identifier StudentID
Partial Key Dashed underline Key within weak entity (needs parent) LineItemNumber

Relationship Cardinality (Modified Chen)

Using Crow's Foot in Chen

Modified Chen notation incorporates crow's foot cardinality symbols:

  • | = One and only one
  • ○ = Zero
  • Crow's foot = Many
  • Double bar || = Must have one and only one (mandatory)

Key Takeaways

🔄 Subtypes
  • Use for "Is A" relationships
  • Exclusive: entity is ONE subtype
  • Overlapping: entity is MANY subtypes
  • Subtypes inherit attributes and relationships
🎭 Roles
  • Clarify entity participation
  • Essential for recursive relationships
  • Use NOUNS not verbs
  • Enable multiple relationships
🔗 Weak Entities
  • Need parent for identity
  • Composite PK includes parent's PK
  • Identifying relationship
  • Not the same as mandatory!
📝 Notation
  • Chen: Rich attributes (learning)
  • IE: Crow's foot (industry)
  • Chen: Double rectangle = weak
  • IE: Solid line = identifying
