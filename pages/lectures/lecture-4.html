Database Views

What is a View?

A view is a virtual table derived from other tables. It does not exist in physical form (unlike base tables where tuples are always stored). Think of it as a stored SQL query that runs on-demand and looks like a table.

Key Characteristics

  • Virtual: Does not store data physically - it's a stored query
  • Read-Only: Generally cannot use INSERT/UPDATE/DELETE (some DBMS support under conditions)
  • Composable: Views can reference other views; queries can join views like tables
  • Dynamic: Data is always current, fetched from underlying tables on query

Creating Views for Security

Example: Limiting Data Access
-- Create a view that excludes sensitive department
CREATE VIEW EmployeeBasicInfo
AS
SELECT EmployeeID, FirstName, LastName, Email
FROM Employee
WHERE department <> 'CIA Internal Investigations';

-- Grant access to view instead of base table
GRANT SELECT ON EmployeeBasicInfo TO Recruiter;

Security Benefit: Instead of granting access to the entire Employee table, we grant access only to the view, which limits both the attributes (columns) and rows visible to the user.

Complex View Example

Example: Sales Analysis Views
-- View: Customer load per sales rep
CREATE VIEW SALES_REP_Customer_Load AS
SELECT s.repnum, s.dept_id, COUNT(*) AS customer_count
FROM Sales_Rep r
LEFT OUTER JOIN Customer c ON r.repnum = c.repnum
GROUP BY s.repnum, s.dept_id;

-- View: Department statistics
CREATE VIEW Dept_Profile(dept_id, staff_count, avg_salary) AS
SELECT dept_id, COUNT(*), AVG(Salary)
FROM DEPARTMENT d
JOIN Sales_Rep r ON d.dept_id = r.dept_id
GROUP BY dept_id;

Views can be joined together just like tables: 

-- Join two views as if they were tables
SELECT l.repnum, l.dept_id, staff_count, avg_salary
FROM SALES_REP_Customer_Load l
JOIN Dept_Profile d ON l.dept_id = d.dept_id
WHERE staff_count > 40;

How Views Stay Current

Technique How It Works Best For
Dynamic Query Query runs on-demand against underlying tables Simple views, real-time data needs
View Materialization Create temporary/permanent table when view is first queried (caching) Complex queries, large datasets
Incremental Updates Materialized view updated automatically or on schedule when related data changes Performance-critical reporting
⚠️ Dropping Views and Dependencies

DROP VIEW EmployeeBasicInfo; only removes the view, not underlying tables.

However, DROP TABLE employee CASCADE; will also drop any views built on that table since they are dependent on it!

CASE Expressions

CASE Statement Basics
  • Use when a value differs based on certain conditions
  • Only one case is expected to be true
  • Evaluation stops when a true case is found
  • Can be used anywhere a value is expected: querying, inserting, or updating

Syntax Structure

CASE WHEN condition THEN result
WHEN condition THEN result
ELSE default_result
END [AS alias]

Example 1: Location Derivation

Categorize customers by postal code
SELECT 
    CASE WHEN postal_code LIKE 'R%' THEN 'MB Customer'
         ELSE 'Non-Local'
    END AS customer_location,
    last_name,
    first_name
FROM customer;

Example 2: NULL Substitution

Replace NULL with meaningful text
SELECT publisher_name,
    CASE WHEN parent_company IS NULL 
         THEN 'No Parent Company'
         ELSE parent_company 
    END AS parent_company
FROM publisher;

Common Use: NULL substitution provides more informative output than displaying blank or "null" values.

Example 3: Conditional UPDATE

Department-based salary raises

"Department 5 gets a 20% raise, department 4 gets 10%, and department 1 gets 30%." 

UPDATE EMPLOYEE
SET Salary = CASE
    WHEN Dno = 5 THEN Salary * 1.2
    WHEN Dno = 4 THEN Salary * 1.1
    WHEN Dno = 1 THEN Salary * 1.3
    ELSE Salary
END;

Example 4: Deriving from Aggregates

Employment status from hours worked
SELECT employee_id, SUM(hours),
    CASE 
        WHEN SUM(hours) = 40 THEN 'FT'   -- Full Time
        WHEN SUM(hours) < 40 THEN 'PT'   -- Part Time
        WHEN SUM(hours) > 40 THEN 'OT'   -- Overtime
        ELSE 'n/a'
    END AS EmploymentStatus
FROM timesheet
GROUP BY employee_id;
💡 Pro Tip

CASE expressions can be nested and combined with aggregate functions, making them extremely powerful for reporting and data transformation tasks.

UNION Operator

What is UNION?

The UNION operator performs set operations on two or more SQL query results, combining them into a single result set.

Two UNION Variations

Operator Behavior Use When
UNION Combines results and eliminates duplicates (like DISTINCT across queries) You need distinct combined results
UNION ALL Combines results without eliminating duplicates You want all rows, or know there are no duplicates
⚠️ Important Requirements
  • Both SELECT statements must have the same number of columns
  • Corresponding columns must have compatible data types
  • Column titles come from the first query in the UNION

Basic Syntax

SELECT column1, column2 FROM table1
UNION
SELECT column1, column2 FROM table2;

-- Or with UNION ALL:
SELECT column1, column2 FROM table1
UNION ALL
SELECT column1, column2 FROM table2;

Performance Consideration

💡 Performance Tip

UNION ALL is more efficient than UNION because it doesn't need to check for and eliminate duplicate rows. Use UNION ALL when you know there are no duplicates or when duplicates don't matter.

PostgreSQL Limitation

⚠️ ORDER BY in UNION (PostgreSQL)

PostgreSQL only allows ORDER BY on the last query in a UNION. Other RDBMS may have different limitations. If you need to sort individual query results before the UNION, use subqueries.

Key Takeaways

👁️ Views
  • Virtual tables - stored queries, not stored data
  • Great for security (limit columns/rows)
  • Can reference other views
  • Materialization can improve performance
🔀 CASE
  • Conditional logic in SQL
  • Stops at first true condition
  • Use for value derivation
  • Common for NULL substitution
➕ UNION
  • Combines multiple query results
  • UNION removes duplicates
  • UNION ALL keeps all rows (faster)
  • Must have same column count/types
